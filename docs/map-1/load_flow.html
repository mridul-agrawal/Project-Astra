Map 1 — the load flow

From the last line of the opening cutscene to a fully-rendered battle map with units. Beat-by-beat through the code.

One sentence: the cutscene stashes which map to load in a shared BattleContext asset → the state machine swaps to the BattleMap scene → MapBootstrapper reads that pending map, paints its tiles, and asks UnitSpawner to instantiate the units the map authored → TurnManager finds them and begins Player Phase.

2CutsceneUI.GoToBattleMap()engine · edited
The seam we added: before asking to change state, it records which map this cutscene leads into, into the shared BattleContext asset.
private void GoToBattleMap()
{
    // _battleMap = Map1_BridgeAtSuvarnapur (wired in the Inspector)
    if (_battleContext != null && _battleMap != null)
        _battleContext.RequestMap(_battleMap);          // remember the map
    GameStateManager.Instance.RequestTransition(GameState.BattleMap, "CutsceneUI");
}
touches BattleContext (sets its pending map) + MapData (Map 1 reference)
▼
3BattleContext.RequestMap()new
A tiny runtime-variable ScriptableObject. It just holds the map in memory — but because it's an asset, that value survives the scene swap that's about to happen.
private MapData _pendingMap;
public void RequestMap(MapData map) => _pendingMap = map;
public MapData Consume() { var m = _pendingMap; _pendingMap = null; return m; }
B · State machine → scene swap
No map data travels through the transition itself — the state machine only flips a state; the map waits in the BattleContext asset.
4GameStateManager.RequestTransition()engine
Validates the move (Cutscene → BattleMap) against the transition table, then raises a state-changed event on a shared channel.
if (IsIllegalTransition(target, requester)) return false;
ExecuteTransition(target);                       // _currentState = BattleMap
_stateChangedChannel.Raise(new StateChangeArgs {
    PreviousState = Cutscene, NewState = BattleMap });
▼
5SceneLoader.OnStateChanged()engine
Subscribed to that channel. It loads the Unity scene named after the enum. The BattleContext asset is untouched by the load, so the pending map carries across.
if (!SceneStates.Contains(args.NewState)) return;
SceneManager.LoadScene(args.NewState.ToString());   // "BattleMap"
the BattleContext asset persists across LoadScene — this is how "which map" crosses the scene boundary
C · BattleMap boots — paint the tiles
Scene objects wake up. MapBootstrapper is forced to run first via [DefaultExecutionOrder(-100)], so the map + units exist before the turn system looks for them.
6MapBootstrapper.Start()engine · edited
Consumes the pending map (falling back to its serialized map for direct editor-play), renders it, then spawns its units.
[DefaultExecutionOrder(-100)]   // before TurnManager
private void Start()
{
    MapData map = _battleContext.Consume();   // Map1, then clears it
    if (map == null) map = _mapData;          // fallback (TestMap16x16)
    _mapRenderer.LoadMap(map);                // → beat 7
    _unitSpawner.SpawnUnits(map);             // → beat 8
}
reads BattleContext (consume) → gets the MapData to use
▼
7MapRenderer.LoadMap()engine
Clears the tilemaps and stamps every cell. Each cell's integer tile-id is looked up in the map's tileset to get a real TileBase, which is set on the Unity Tilemap.
_currentMap = mapData; ClearAllTilemaps();
// per layer, per (x,y):
int tileId = mapData.GetTileId(layer, x, y);   // flat tileIds[y*W + x]
TileBase tile = tileset.GetTile(tileId);      // id → TileBase
tilemap.SetTile(new Vector3Int(x, y, 0), tile);
reads MapData.layers[].tileIds + TilesetDefinition (see "How a map stores its tiles")
D · Spawn the units from data
The heart of the new unit system — units are no longer hand-placed; they come straight out of the map's authored list.
8UnitSpawner.SpawnUnits()new
If the map authored any units, it clears whatever was in the scene and instantiates one GameObject per authored position. Maps with an empty list are left alone (the old test map keeps its hand-placed units).
public void SpawnUnits(MapData map)
{
    if (map.UnitStartPositions.Length == 0) return;   // legacy maps untouched
    ClearExistingUnits();                             // data map owns its roster
    foreach (var start in map.UnitStartPositions)
        SpawnUnit(start);                             // → beat 9
}
reads MapData.unitStartPositions = { position, unitId, team }
▼
9UnitSpawner.SpawnUnit() + UnitDatabasenew
Resolve the string unitId → a UnitDefinition asset, build the GameObject, set its faction (from the team number) and grid position, give it the definition + its map sprite.
_unitDatabase.TryResolve(start.unitId, out UnitDefinition def);  // "aranya" → asset
var go = new GameObject(def.UnitName);
var unit = go.AddComponent<TestUnit>();          // auto-adds UnitInventory

E · The turn system picks them up
Because the bootstrapper ran first, the units already exist when the turn manager scans the scene.

How a map stores its tiles

Tile pipeline — an int becomes a painted tile

MapData.layers[].tileIds[ ]
flat ints, row-major → tilesetIndex → MapData.tilesets[ ] → TilesetDefinition._tiles[id]
{ TileBase, TerrainType } → Tilemap.SetTile(x,y)
Map 1's grid is 11×11 = 121 ints. id 0=plains, 1=water, 2=bridge, 3=forest, 4=hut. The same TerrainType on each entry is what movement/combat read (water/forest/hut = impassable). North is stored at high y (mockup row 0 → grid y = 10), so Unity's y-up renders it the right way round.

How a map stores its units

Unit pipeline — an authored row becomes a living unit

MapData.unitStartPositions[ ]
{ position, unitId, team } → UnitDatabase.TryResolve(unitId) → UnitDefinition
class · stats · mapSprite · isLord → TestUnit + UnitInstance
live HP / move / faction
Map 1 authors 6 rows: aranya@(8,2) team 0, rakshasa_raider@(5,7) & rakshasa_miniboss@(5,9) team 1, three villager team 2. The team int maps to the faction; the unitId string is the only link between the map and the character data.

The two seams this added

Everything else already existed — we inserted exactly two joints

  • BattleContext (+ optional MapCatalog) — lets anything (cutscene, menu, future world-map) say "load this map next" at runtime, instead of a hardcoded Inspector field. The map travels across the scene swap inside this asset.
  • UnitSpawner (+ UnitDatabase + UnitDefinition.mapSprite) — turns MapData.unitStartPositions (which existed but was never read) into actual units, so units travel with the map.

Scripts new in this work: BattleContext, MapCatalog, UnitDatabase, UnitSpawner. Edited: MapData (+mapId), UnitDefinition (+mapSprite), MapBootstrapper, CutsceneUI, TestUnit. The [DefaultExecutionOrder(-100)] on MapBootstrapper is what guarantees beats 6–10 finish before beat 11.
